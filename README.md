# 🎓 CampusLoyalty — Платформа мотивации студентов

Закрытая B2B2C-платформа для управления внутренней валютой и мерчем в университете. Деканат и кураторы геймифицируют учебный процесс, начисляя валюту за активности. Студенты тратят её на эксклюзивный мерч (общий и факультетский).

> **Цель проекта:** демонстрация навыков проектирования RBAC, кастомных сценариев безопасности, отказоустойчивых хранилищ и изоляции данных на стеке Spring Boot.

---

## 💼 Бизнес-логика

### 👥 Ролевая модель (RBAC)

| Роль | Права | Видимость |
|------|-------|-----------|
| **DIRECTOR** | Полный доступ, управление кураторами | Всё |
| **CURATOR** | Создание квестов, назначение старост, управление факультетом | Свой факультет |
| **STAROSTA** | Помощник куратора, начисление баллов | Своя группа |
| **STUDENT** | Участие в квестах, покупка мерча | Только доступные квесты |

### 🎓 Жизненный цикл студента

Пользователи не регистрируются сами — их добавляет деканат при зачислении. Статус отражает положение студента в университете:

- **ACTIVE** — зачислен, полный доступ
- **ACADEMIC_LEAVE** — в академе, ограниченный доступ
- **DISMISSED** — отчислен, доступ заблокирован
- **GRADUATED** — выпускник, ограниченный доступ

### 🏷️ Изоляция данных

- **Общие квесты/мерч** (от декана) — видны всем
- **Факультетские квесты/мерч** (от куратора) — видны только студентам этого факультета

Отказоустойчивое хранилище токенов
Система использует двухуровневое хранилище для токенов установки пароля:
┌─────────────────┐
│  Генерация      │
│  токена         │
└────────┬────────┘
         │
         ▼
┌─────────────────┐     Успех   ┌──────────────┐
│  Сохранить в    │────────────▶│ Redis        │
│  Redis (TTL 24ч)│             │ (основное)   │
└────────┬────────┘             └──────────────┘
         │ DataAccessException
         ▼
┌─────────────────┐
│  Сохранить в    │
│  PostgreSQL     │
│  (fallback)     │
└─────────────────┘

Почему такое решение?
Redis: быстрый доступ (in-memory), автоматическое удаление по TTL (24 часа)
PostgreSQL: надежность — работает, даже если Redis упал
Безопасность
Пароли: хэшируются через BCrypt, никогда не хранятся в открытом виде
JWT: подписываются секретным ключом (HS256), содержат ID пользователя и роли
Токены: UUID, одноразовые, удаляются после использования
CSRF: отключен (не нужен для REST API с JWT)
Сессии: Stateless, сессии не создаются на сервере

🔐 Защита персональных данных
Телефонные номера студентов шифруются перед сохранением в БД через DataCodecService:
Flow:
Пользователь вводит номер (например, +7 (999) 123-45-67)
Сервис нормализует: 79991234567
Шифрует через обфускацию (сдвиг символов + флаг реверса)
Сохраняет в БД зашифрованный вид
При возврате клиенту — дешифрует обратно
Зачем:
Защита от случайного чтения при утечке БД
Соответствие требованиям по защите персональных данных
Примечание: для production рекомендуется использовать AES вместо обфускации.
### 🔍 Поиск пользователей (паттерн Strategy)

Поиск реализован через паттерн **Strategy** — каждая стратегия отвечает за свой тип данных:

| Стратегия | Формат ввода | Regex |
|-----------|--------------|-------|
| `UserLastNameSearchStrategy` | Фамилия (буквы, пробелы, дефис) | `[a-zA-Zа-яА-ЯёЁ\s-]+` |
| `UserPhoneNumberSearchStrategy` | Телефон (цифры) | `\d+` |
| `UserTicketSearchStrategy` | Студенческий билет (XXX-XXX) | `\d{3}-\d{3}` |

Как работает:
Пользователь вводит строку поиска
Система перебирает стратегии и вызывает isApplicable(input)
Первая подходящая стратегия выполняет поиск через search(input)
Особенности:
Телефон шифруется через DataCodecService перед поиском (соответствует формату хранения в БД)
Поиск по фамилии — без учета регистра
Все стратегии внедряются списком через Spring — легко добавлять новые

### Глобальный обработчик

`SchoolControllerAdvice` перехватывает все исключения и возвращает структурированные ответы через DTO `SchoolError`:

| Тип исключения | HTTP-статус | Пример |
|----------------|-------------|--------|
| `AuthException` | 400, 403, 409 | Неверный токен, доступ запрещен |
| `BadCredentialsException` | 401 | Неверный email/пароль |
| `NotFoundException` | 404 | Сущность не найдена |
| `BalanceOperationException` | 409, 422 | Недостаточно средств |
| `MethodArgumentNotValidException` | 400 | Ошибка валидации DTO |
| `BindException` | 400 | Невалидные query-параметры |
| `HttpMessageNotReadableException` | 400 | Невалидный JSON в теле запроса |
| `HttpRequestMethodNotSupportedException` | 400 | POST вместо GET |
| `MissingServletRequestParameterException` | 400 | Отсутствует обязательный параметр |
| `DataAccessException` | 500 | Ошибка БД/Redis |
| `Exception` | 500 | Все остальные ошибки |

Структурированный ответ
Все ошибки возвращаются в едином формате:

{
  "code": "INVALID_ACTIVATION_TOKEN",
  "message": "Ссылка для активации недействительна, либо её срок действия (24 часа) истек",
  "timestamp": "2026-07-10T14:30:45.123"
}

Централизованное хранение сообщений
Все текстовые сообщения вынесены в классы констант, разделенные по аудиториям:
AppConstants — общие константы (Validation, Balance, SystemErrors, NotFound)
AuthConstants — константы модуля авторизации:
Success — успешные ответы для клиента (фронтенд)
Errors — сообщения об ошибках для пользователя на экране
Logs — технические сообщения для логов разработчика
Преимущества:
Изменение текста в одном месте обновляет его везде
Разделение UI-текстов и технических логов упрощает локализацию

🏗 Ключевые технические решения
Почему не OAuth2/Keycloak?
Бизнес-требование: студенты не регистрируются сами, база пользователей импортируется из деканата. Использование тяжелых IAM-систем избыточно. Реализован кастомный, но безопасный онбординг.
Почему двухуровневое хранилище токенов?
Redis быстрый, но может упасть. PostgreSQL надежный, но медленный. Комбинация дает и скорость, и отказоустойчивость.
Как решена изоляция данных по факультету?
На уровне репозиториев (Spring Data JPA) реализована фильтрация. Студент видит только квесты/мерч своего факультета и общие.
Почему BigDecimal для баланса?
Избегаем ошибок плавающей точки при финансовых операциях. Точность 10 цифр, 2 знака после запятой.
Почему паттерн Strategy для поиска?
Разные форматы ввода (фамилия, телефон, билет) требуют разной логики. Strategy позволяет легко добавлять новые типы поиска без изменения существующего кода.
